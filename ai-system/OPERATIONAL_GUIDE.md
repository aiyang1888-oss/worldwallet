# 🚀 AI Security Audit System - 运营指南

**阶段**: 运营阶段（使用系统常态化）  
**目标**: 持续发现问题 → 持续降低风险  
**重点**: 使用频率 + 规则积累 + 风险控制

---

## 📊 标准运行机制

### 1️⃣ 定期审计（Daily + Pre-Commit）

#### 日常审计（每日）
```bash
# 每天固定时间运行
node ai-system/audit-mode.js

# 输出物
# ├─ WORLDWALLET_SECURITY_REPORT.md
# ├─ ai-system/reports/security-report-YYYY-MM-DD.json
# └─ Terminal 摘要 + Telegram 通知
```

**时间建议**: 每天 08:00 UTC+7（团队早会后）

#### 提交前审计（必须）
```bash
# 任何重要代码变更前必须运行
git status  # 查看待提交代码
node ai-system/audit-mode.js  # 运行审计
# 检查报告，确认无新增 HIGH 风险
git commit -m "..."
```

**触发条件**:
- [ ] security.js 有改动 → 必须审计
- [ ] 涉及加密/解密代码 → 必须审计
- [ ] 密钥/助记词处理 → 必须审计
- [ ] 其他代码变更 → 建议审计

---

### 2️⃣ 问题转任务（闭环关键）

#### 发现问题后的流程
```
审计报告 
  ↓
识别 HIGH/MEDIUM 风险
  ↓
创建 GitHub Issue
  ↓
分配开发人员
  ↓
修复并测试
  ↓
重新审计验证
  ↓
关闭 Issue
```

#### Issue 模板（复制使用）

```markdown
**标题**: [Security] Fix {SECURITY-ID}: {描述}

**优先级**: HIGH / MEDIUM / LOW

**问题描述**:
{从审计报告中复制}

**文件位置**:
- 文件: {文件路径}
- 行号: {行号}

**代码片段**:
{从审计报告中复制}

**修复建议**:
{从审计报告中复制}

**验证方式**:
1. 修复代码
2. 运行: npm run test
3. 重新审计: node ai-system/audit-mode.js
4. 确认风险已消除

**修复人员**: @{assigned}
**截止日期**: {根据优先级设定}
```

#### 风险级别对应的处理时间

| 级别 | 修复期限 | 说明 |
|------|---------|------|
| **HIGH** | 3 天内 | 严重风险，立即修复 |
| **MEDIUM** | 1 周内 | 需要修复，排入计划 |
| **LOW** | 2 周内 | 可以修复，按优先级处理 |

---

### 3️⃣ 审计历史沉淀（资产化）

#### 历史记录保留
```
ai-system/reports/
├── security-report-2026-04-08.json    # 日期 1
├── security-report-2026-04-09.json    # 日期 2
├── security-report-2026-04-10.json    # 日期 3
└── ...（持续保留）
```

**不删除**：所有历史报告永久保存（便于追溯）

#### 定期分析（每月）

```bash
# 生成月度总结
node ai-system/monthly-summary.js 2026-04

# 输出：
# ├─ HIGH 风险趋势
# ├─ 问题重复情况
# ├─ 修复完成率
# └─ 规则覆盖度分析
```

#### 统计维度

| 维度 | 跟踪指标 | 目标 |
|------|---------|------|
| **风险数量** | HIGH / MEDIUM / LOW 趋势 | 持续下降 |
| **重复风险** | 同一风险是否再次出现 | 0 重复 |
| **修复率** | 发现的问题中有多少被修复 | >95% |
| **规则覆盖** | 当前规则能覆盖多少模块 | 逐月增加 |

---

### 4️⃣ 规则库扩展（核心长期价值）

#### 当前规则（起点）
```
✅ atob() 错误处理
✅ Hardcoded salt 检查
✅ Direct comparison for sensitive data
```

#### 待扩展规则库（沉淀新问题为规则）

| 规则 ID | 描述 | 优先级 | 状态 |
|---------|------|--------|------|
| SECURITY-004 | 加密函数缺少初始化向量（IV）验证 | HIGH | ⏳ |
| SECURITY-005 | 私钥存储在内存中未清零 | HIGH | ⏳ |
| SECURITY-006 | 助记词处理无加密存储 | HIGH | ⏳ |
| SECURITY-007 | 不安全的随机数生成 | MEDIUM | ⏳ |
| SECURITY-008 | 明文日志包含敏感数据 | MEDIUM | ⏳ |
| SECURITY-009 | 输入验证缺失（base64/hex） | MEDIUM | ⏳ |
| SECURITY-010 | 无效数据处理缺少防御 | LOW | ⏳ |

**规则创建流程**:
```
发现新问题
  ↓
分析根因
  ↓
创建扫描规则（scout-security.js）
  ↓
在全项目范围内扫描
  ↓
验证规则有效性
  ↓
加入规则库
```

## 🔒 运营原则

1. **发现为主**
   - 系统的工作是发现问题
   - 人负责决定如何处理

2. **可追溯性**
   - 所有审计结果都有记录
   - 所有修复都可追溯

3. **持续迭代**
   - 发现的问题 → 规则
   - 规则 → 全面覆盖

4. **风险导向**
   - 优先处理 HIGH 风险
   - 逐步降低总体风险
